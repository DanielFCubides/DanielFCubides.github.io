# CLAUDE.md

## Project Overview
This is a personal Hugo static site (DanielFCubides.github.io) — a blog and portfolio for a software engineering leader.

## Stack
- **Hugo** static site generator with the **Ananke** theme (git submodule at `themes/ananke/`)
- Deployed to **GitHub Pages** via the `docs/` folder (built output)
- Content written in **Markdown**

## Key Commands
```bash
hugo server -D          # Local dev server at localhost:1313 (includes drafts)
hugo                    # Build production site into docs/
```

## Content Structure
- `content/posts/` — Blog posts (Markdown)
- `content/cv/` — CV/resume page
- `content/_index.md` — Homepage content
- `static/` — Static assets (images, PDFs)
- `config.yaml` — Site configuration

## Writing New Posts
Use Hugo's archetype system:
```bash
hugo new posts/my-post-title.md
```
Posts start as `draft: true` — set to `false` to publish.

## Front Matter Convention
```yaml
---
title: "Post Title"
date: YYYY-MM-DDTHH:MM:SS-05:00
draft: false
---
```

## Target Audience
Posts are written for **junior software engineers** who are learning about software architecture and software engineering practices. Assume the reader:
- Has basic programming knowledge but limited industry experience
- Is unfamiliar with advanced architectural patterns and concepts
- Benefits from concrete examples, analogies, and step-by-step explanations
- May encounter technical jargon for the first time

Always define acronyms and technical terms on first use. Avoid assuming prior knowledge of advanced concepts.

## Tone & Writing Style
- **Professional but explanatory** — write with authority and clarity, but prioritize understanding over impressiveness
- Break down complex concepts into digestible steps before going deeper
- Use real-world analogies to make abstract ideas tangible
- Prefer short paragraphs and numbered/bulleted lists for readability
- First-person voice is encouraged
- Avoid unnecessary jargon; when technical terms are needed, explain them in context
- End posts with a clear takeaway or summary of key lessons

## Content Guidelines
- Blog topics: software architecture, engineering leadership, team culture, technical practices
- Posts are in English
- Do NOT edit files inside `themes/ananke/` — it is a git submodule
- Do NOT edit files inside `docs/` manually — they are generated by `hugo`
- Do NOT edit files inside `resources/_gen/` — auto-generated cache

## Deployment
1. Run `hugo` to build into `docs/`
2. Commit and push to `main` branch
3. GitHub Pages serves from `docs/` folder automatically

## What NOT to Do
- Do not modify the `themes/ananke` submodule
- Do not manually edit `docs/` (always regenerate with `hugo`)
- Do not add unrelated dependencies or config files
